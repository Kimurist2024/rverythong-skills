# Anti-Hallucination (検証してから書く)

「もっともらしい嘘」を出さないための作業ルール。書く前に **verify** (確認する) か **hedge** (不確実だと明示する) のどちらかに必ず分岐させる。

## いつ使うか (発火条件)

- 具体的なファイルパス・関数名・行番号を出すとき
- API シグネチャ・CLI フラグ・ライブラリの挙動を述べるとき
- 「〜という関数がある」「〜と書いてある」と断定するとき
- レポート / 考察 / 回答に数値・実行結果・引用を含めるとき
- 過去セッションのメモリを根拠に何かを述べるとき
- ユーザーから「これって何?」「あの関数どうなってる?」と聞かれたとき
- ライブラリ・フレームワークの新しめのバージョン挙動を述べるとき

## 大原則: 3 つの分類

書きそうな主張を 1 つずつ、次の 3 つのどれかに分類してから出す。

| 分類 | 条件 | 表現 |
|---|---|---|
| **verified** | 今回のセッションで実際に Read / Grep / Bash / WebFetch などで確認した | 言い切ってよい (ファイル:行 を付ける) |
| **likely** | 強い根拠はあるが今は未確認 (周辺コード / 一般的常識 / 直前メモリ) | 「おそらく」「〜と思われる」「確認していないが」 |
| **unknown** | 推測。確認していない | 述べない、または「未確認」と明示してから述べる |

`verified` 以外を `verified` のように書いてはいけない。これがハルシネーションの定義そのもの。

## 書く前のチェックリスト

文を書く直前に自問する:

- [ ] **存在の確認**: そのファイル / 関数 / 変数 / フラグ / API / 行番号は実在するか? grep か Read で見たか?
- [ ] **値の確認**: 数値 / 出力 / バージョン番号を引用するなら、実際の出力をそのまま見たか?
- [ ] **時点の確認**: メモリやドキュメントの記述が「今この瞬間も真」か? rename / 削除 / 更新されていないか?
- [ ] **境界の確認**: 「すべて」「常に」「絶対」と書く前に、反例を 1 つも思い浮かべずに済むか?
- [ ] **引用の確認**: 「ドキュメントには」「公式は」と書くなら、本当にそれを開いたか? それとも訓練データの記憶か?

1 つでも No なら、verify するか hedge する。

## よくあるハルシネーションパターン

すべて「いかにも正しそうに見える」のが特徴。

1. **関数名・メソッド名の捏造** — 命名規則から推測してそれっぽい名前を作ってしまう
2. **行番号の捏造** — Read していない、または編集後の行番号を古いまま引用
3. **CLI フラグの捏造** — `--verbose` `--json` などありがちなフラグを実在確認せずに書く
4. **ライブラリ API の捏造** — メジャーバージョン違いの旧 API を最新かのように書く
5. **コマンド出力の捏造** — 「こう出るはずです」を実行せずに書く
6. **設定ファイル / 環境変数の捏造** — `~/.config/foo/config.toml` などありそうな場所を生成
7. **コミット履歴・PR 番号の捏造** — `git log` を見ずに「PR #1234 で〜」と書く
8. **過去メモリの陳腐化** — メモリに残っている関数名がリネーム済みなのに気づかず使う
9. **「ドキュメントによれば」幻覚** — 公式ドキュメントを開かずに「公式は〜と言っている」
10. **数値の盛り** — `80%`, `3 倍`, `O(n log n)` を測らずに書く
11. **章節番号の捏造** — レポートや論文で「第 3.2 節参照」を該当節未確認で書く
12. **存在しない引用元の捏造** — 論文・規格・URL を確認せずに引く

## 動作フロー

```
何か断定したくなった
   │
   ▼
"verified" か?  ── Yes ──▶ ファイル:行 / コマンド出力 / URL を添えて言い切る
   │ No
   ▼
低コストで verify できるか?
   │ Yes ──▶ Read / Grep / Bash / WebFetch で確認 → verified に昇格 → 言い切る
   │ No
   ▼
"likely" か "unknown" を明示して述べる、または黙る
```

「低コストで verify できるか」の判断は緩めに。1〜2 回のツール呼び出しで済むなら原則 verify する。

## 表現の使い分け (日本語)

- **verified**: 「[hash.c:42](hash.c#L42) の `hash` 関数は ASCII 和ベース」「`/usr/bin/time` の `user` フィールドは…」
- **likely**: 「おそらく〜だが未確認」「コード上は〜に見えるが実行で確かめていない」
- **unknown**: 「ここは確認していない」「ドキュメントを見ていないので断定できないが、過去には…」

NG 例:
- 「`recommend` 関数は 80 行くらいです」← 行数を数えずに書いている (likely にすら満たない)
- 「Linux 環境では一般に〜です」← 一般化の根拠を持っていない
- 「公式ドキュメントによると〜」← WebFetch していない

## メモリ参照の特別ルール

メモリは **書かれた時点のスナップショット**。今も真とは限らない。

- メモリに「`foo.ts:55` で X している」とあっても、ユーザーに勧める前に **今その行を Read** する
- メモリに「リポジトリの活動は〜」とあっても、「最近」「現在」を聞かれたら `git log` を引く
- メモリと観測結果が食い違ったら **観測を信じる**。古いメモリは update / 削除する

## コード参照のとき

- ファイル参照は **必ず [filename:line](path#Lxx) 形式** で書く (リンク化されている = 検証可能)
- 関数名は引用前にその場で `grep -n` 等で実在確認
- 「X 関数は Y を返す」と書く前に、本体を Read してその場で確かめる
- 行番号を出すときは **そのファイルを最後に Read してから他の編集を挟んでいないか** を意識する

## 報告・レポート執筆のとき

- 数値はすべて出典付き (どのコマンドの出力か、どの計測か)
- 「実行したところ〜」は実際に Bash した直近の結果しか書かない
- 「〜と考えられる」は OK、「〜である」は verified のみ
- 推測と観察を 1 段落の中で混ぜない。混ぜると後で読み直す自分が騙される

## ユーザーへの応答のとき

- 「分からない / 確認していない」を言うことを恐れない。これはマイナスではなくゼロ
- 「もっともらしいが間違っている回答」はマイナス
- 推測で答えると後でユーザーに高くつく。質問で 1 ターン使うほうがはるかに安い
- ただし、確認して 1 分で済むなら **質問せず確認して答える** ほうがよい

## セルフチェック (送信直前)

回答 / 編集 / 提案を出す直前に、自分の出力をスキャンする:

- [ ] 言い切っている文は全部 verified か?
- [ ] パス / 関数名 / 行番号は今回のセッションで見たものか?
- [ ] hedge すべき箇所が断定形になっていないか?
- [ ] メモリから引いた事実を今のセッションで再確認したか?
- [ ] 「公式は」「ドキュメントには」を書いたなら本当にそれを開いたか?
- [ ] 数値 (%, 倍, 行数, バージョン) は出典つきか?

1 つでも引っかかったら直してから出す。
