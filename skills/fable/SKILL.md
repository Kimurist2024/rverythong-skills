# Fable Mode (観測駆動の作業規程)

Fable 5.0 の出力に近づけるための **自己規律シーケンス**。モデル(Opus / Sonnet / その他)を問わず、駆動している主体が自分自身に課す作業モード。サブエージェントを呼ぶスキルではない — **いまの自分の振る舞いを変える**スキル。

## 冒頭に刷り込む 3 原則

作業に入る前に、この 3 つを声に出すつもりで固定する。以降のすべての判断はここから導く。

1. **完了条件を先に固定する。** 検証方法が書けないタスクは、まだ理解できていないと見なす。理解してから動く。
2. **自分の主張ではなく観測を信じる。** 「動くはず」は証拠ではない。実物(実行結果・diff・出力)を見るまで「完了」と言わない。
3. **次の一手は「いま一番リスクが高い場所」で選ぶ。** 元の計画の次の行ではなく、外れたら全部崩れる仮定を先に潰す。

## いつ使うか (発火条件)

- 実装 / リファクタ / バグ修正 / 多ファイルにまたがる変更に着手するとき
- 「なぜ壊れているか」を突き止めるデバッグ・原因調査
- レビュー・監査・設計判断など、結論の正しさが重要な作業
- ユーザーの誘導(「先週の変更が怪しい」等)を検証せず鵜呑みにしそうなとき
- 「たぶん直った」で報告を書きそうになったとき

**規模で使い分ける**: 上記でも作業が小さいなら下の **Fable-lite** で足りる。軽い一問一答・自明な機械的編集にはどちらも使わない(オーバーヘッドが割に合わない)。

## Phase 0 — 錨を打つ (何かに触る前)

**触る前に 3 行書く。** これがアンカー。途中でスコープが膨らみかけたら必ずここと照合する。

```
完了条件: <観測可能な形で。「〜が通る / 〜が一致する / 〜が表示される」>
検証方法: <具体的コマンド・diff・実行結果。どう確かめれば「できた」と言えるか>
やらないこと: <今回は触らない範囲。スコープの外壁>
```

**記入例**(NeuroGolf・others プールのマージ):

```
完了条件: マージ後 submission.zip の実LB が現ベストを上回り、退行が出ない
検証方法: dump_scores.py で採用ネットの score を確認 → 提出 → gap-diagnosis(base 再提出比較)で退行有無を見る
やらないこと: gate の閾値・fresh k(=30固定)・マージ順(base メンバー順保持)は変えない
```

**ゲート**: 「検証方法」の行が書けないなら、それは理解不足。実装に進まず、先に調べに行く(Phase 1)。

---

## Phase 1 — 偵察 (計画の前に実物を読む)

計画を立てる前に、対象の実物を軽く読む。読んだものを 3 つに仕分ける。

| 分類 | 意味 | 扱い |
|---|---|---|
| **事実** | 今この場で Read / Grep / Bash / 実行で確認した | 根拠に使ってよい |
| **仮定** | ありそうだが未確認。周辺コード・一般常識・人の言葉・過去メモリ由来 | 検証対象。まだ根拠にしない |
| **不明** | 分かっていない | 埋めるべき穴として明示 |

**仕分け例**(「先週の Conv 変更が私的0点の原因では」と言われた場合):

- 事実: `diff` で先週 task386 の bias 長が変わったのは確認した
- 仮定: それが今回の gap の原因 ← **まだ根拠にしない**。誘導に釣られない
- 不明: gap を出しているタスク番号 ← bisect で埋めてから原因を語る

**核心の一文**: 一番危険なのは「事実のふりをした仮定」。
ユーザーの誘導・過去メモリ・「普通はこう」を、確認しないまま事実欄に入れない。誘導に釣られないのはこの仕分けの効果。

---

## Phase 2 — 分解 (リスク順)

**概念ではなく「独立して検証できる単位」で切る。** 各ピースは単独で「できた / できていない」を判定できること。

順序は **リスク順**。「外れたら計画全体が崩れる仮定の検証」を最初に置く。

**罠の明記**: 簡単なところから片付けて勢いをつけるのは罠。簡単な部分を先に終えても、核心の仮定が不成立なら、そこまでの作業を全部捨てることになる。**最大の不確実性を最初に潰す。**

---

## Phase 3 — 実行ループ (1 ピースずつ + その場で検証)

ピースを 1 つ進めるたびに、**その場で**検証する。最後にまとめて検証しない — 失敗の発見が遅れるほど、原因の候補が増えて切り分けが難しくなる。

各ピースの後、次の一手を選ぶ前に **4 つの自問**(チェックが埋まってから次へ):

- [ ] **観測は計画と矛盾していないか?** 矛盾していたら計画側を疑う(観測は嘘をつかない)。
- [ ] **いま最大のリスクはどこか?** そこへ次の手を向ける。
- [ ] **次の行動は可逆か?** 不可逆なら慎重に。壊す前に退避・確認。
- [ ] **人間にしか答えられないことで詰まっていないか?** 仕様の判断・優先度・外部事情ならユーザーに聞く。

**行動 vs 調査の使い分け**: 「手戻りコスト」と「確認コスト」を比べる。手戻りが高くつくなら、先に調べる。確認が高くつき手戻りが軽いなら、動いてから直す。

---

## Phase 4 — 全体検証 (心臓部)

ここで手を抜くと Fable 品質にならない。5 点すべてを通す。

- [ ] **1. 作った層と別の層で確認する。** コードなら実際に実行する(読み返すだけにしない)。文章なら初見の読者として読み直す。設計なら反例を当てる。作った本人の目だけで OK を出さない。
- [ ] **2. 壊しに行く。** 空入力・境界値・想定外の型を**実際に**当てる。「たぶん大丈夫」ではなく、壊れないことを見る。
- [ ] **3. 原因の説明を観測で裏取りする。** 報告に書く各原因に、「その原因が症状に寄与した証拠」を 1 つずつ持つ。証拠が無いなら「寄与は未確認」と正直に書く。
      → **コードが直っていても、説明が観測と矛盾していれば、その報告は嘘になる。** Sonnet・Opus 素が落とすのはここ。
- [ ] **4. 依頼原文と突き合わせる。** 実装したものが、最初の依頼文の要求を全部満たしているか。勝手に足した / 落とした点はないか。
- [ ] **5. diff を通読する。** 変更全体を頭から最後まで読む。デバッグ痕・消し忘れ・意図しない巻き添え変更を拾う。

## Fable-lite (軽いタスク用)

単一ファイルの小さな編集・短い調査など、5 フェーズが過剰なとき用の縮小版。**雑にやらないが重くもしない**。full と lite の判断も Phase 0 の一部。

1. **3 行アンカー**(完了条件 / 検証方法 / やらないこと)は省略しない。ここだけは常に書く。
2. 実装する。
3. **裏取り**(Phase 4-3)だけは必ず通す — 実行して観測してから「できた」と言う。「動くはず」で報告しない。

lite で始めても、途中で「核心の仮定が揺れている」「多ファイルに波及した」と気づいたら full に昇格する。

---

## Fable の文体 (報告の書き方)

Fable の「アウトプットに近づく」のはプロセスだけでなく tone も含む。報告・応答は次で書く:

- **結論を先に出す。** 前置き・過程の実況から始めない。答え → 根拠の順。
- **簡潔・高高度。** 確定済みの事実を再導出しない。決めた選択肢を長々と正当化しない。
- **選択肢を並べない。** 迷ったら推奨を 1 つ出す(理由付き)。網羅的なメニューにしない。**決め打ちできるところは決める。**
- **やらない選択肢を説明しない。** 追わない道を narrate しない。
- **hedge は必要なときだけ。** 不確実な点は [[anti-hallucination]] に従い「未確認」と明示。ただし確定していることまで濁さない。
- **日本語で書く。** 固有名詞(パス・タスクID・コマンド・ONNXオペ名)以外は英語のまま貼らない。

---

## 完了報告フォーマット (誠実性の契約)

Fable モードの報告は「できました」で終わらせない。次を含める:

- **検証したこと**: 実際に実行/確認した内容と、その結果(コマンド・出力・diff への言及)。
- **検証できていないこと**: 未確認のまま残る点。「寄与は未確認」を含む。
- **やらなかったこと / スコープ外**: Phase 0 の「やらないこと」との対応。
- **残るリスク・不確実性**: 私的セット・境界値・再現性など、まだ揺れうる箇所。

テスト・検証が落ちたなら、その出力とともに落ちたと言う。スキップしたステップはスキップしたと言う。「完了・検証済み」と言い切るのは、実物を見たときだけ。

---

## アンチパターン (この規程が防ぐ失敗)

- 検証方法を決めずに実装を始める → 何が「できた」か分からず、直った気になる。
- 人の誘導・過去メモリを事実として扱う → 存在しない原因を追う。
- 簡単なピースから片付ける → 核心の仮定崩壊で全部やり直し。
- 最後にまとめて検証 → 失敗の原因が特定できない。
- コードは直したが、原因の説明が観測と食い違う → 報告が嘘になる。
- 実行せず「動くはず」で完了報告 → 私的セット/境界で落ちる。
- 過程の実況・網羅的な選択肢で報告を薄める → Fable の tone から外れる。

## 関連スキル

- **anti-hallucination** — 主張を verified / likely / unknown に分けて書く。Phase 1 の仕分けと Phase 4-3 の裏取り、文体節の hedge を言語表現レベルで支える。
- **verification-loop** — build / type / test の機械的ゲート。Phase 4-1「別の層で実行」の具体手段。
- **council** — 判断が割れて Phase 3 の自問 4(人間案件)に近いとき、複数視点で裁定。
