# 分析チェックリスト(Phase 1–3 の詳細)

## Phase 1 で確認するファイル(存在する場合は最低限)

- `AGENTS.md` / `CLAUDE.md` / README / プロジェクト固有の指示ファイル
- package・依存関係の設定(package.json / pyproject.toml / go.mod / Cargo.toml 等)
- build / lint / test / typecheck / CI 設定
- docs / spec / 設計メモ
- エントリーポイント・主要モジュール・テスト
- DB schema / migration
- 認証・課金・通知・外部 API・ジョブ・キュー・ストレージ関連
- 既存スクリプトや運用手順

## Phase 1 で整理する内容

1. このプロジェクトは何をするものか
2. 主要なユーザー体験・ワークフローは何か
3. 主要なエントリーポイントはどこか
4. 主要モジュールと責務は何か
5. データはどう流れるか
6. 外部依存は何か
7. 現在の検証コマンドは何か(実在確認 + 実測。発明しない)
8. 絶対に壊してはいけない既存挙動は何か
9. 実行環境の前提(runtime バージョン / package manager / env vars /
   DB・Docker・外部サービスの起動前提)

## Baseline の実測規律

- 安全に実行可能な読み取り系コマンド(lint / typecheck / test)は分析時に実際に実行し、
  `verified: executed-green / executed-failing / defined-only` を付記する
- baseline 時点で失敗しているテストは failure signature とともに「既知の失敗」として
  指示書に列挙する(実装担当・監視者が劣化と誤認しないため)
- 各コマンドに proceed condition(baseline が red のとき続行可か停止か)を定める

## Phase 2: 負債の観点(14)

重複 / 死コード / 責務の混在 / 所有範囲の曖昧さ / 過剰・不足している抽象化 /
依存方向の乱れ / 型・schema・契約の曖昧さ / テスト不足や壊れやすいテスト /
エラーハンドリングやログの不統一 / 設定・環境差分の複雑さ /
非同期・並行処理の危険箇所 / パフォーマンスリスク /
セキュリティ上慎重に扱うべき境界 / 命名・ファイル配置の分かりにくさ

## 「提案のみ(propose only)」にすべき変更

- 公開 API・DB schema・保存データに影響しうる
- 認証・課金・通知・外部連携に影響しうる
- 互換性を壊す可能性がある
- 複数の設計案がありプロダクト判断が必要

## Phase 3: 必ず質問にする条件

- 正しい仕様がコードから判断できない
- テストと実装が矛盾している
- 削除候補のコードが本当に不要か分からない
- 上記「提案のみ」基準のいずれかに該当する変更を実装したい場合

逆に、コードから判断できるものは質問しない(質問の濫造は指示書の信頼を下げる)。
